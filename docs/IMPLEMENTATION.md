## Phase 1: Data Preparation (1 hour)

### Objective
Convert raw 10-K PDF into structured, indexed chunks with parent-child hierarchy and comprehensive metadata.

### 1.1 Understanding the Input

**Expected Structure:**

```
  └─ Additional Tables
```

### 1.2 Create Semantic Chunking Module

#### File: `src/chunking/semantic_chunker.py`

**Responsibility:** 
- Detect structural boundaries in PDF (Item 1, 1A, 8, etc.)
- Split at headers, not arbitrarily
- Preserve section context

**Input:**
- Raw PDF text from PyPDFLoader
- Configuration: chunk_size=512 tokens, overlap=50

**Output:**
- List of chunk dictionaries with content + page markers
- Example: `{"content": "...", "page": 45, "section": "Item 1A", "chunk_idx": 3}`

**Algorithm Steps:**
1. Load PDF using PyPDFLoader (existing in setup_db.py)
2. Detect section headers using regex patterns:
   - Pattern: r"^Item\s+\d+[A-Z]?:" (matches "Item 1:", "Item 1A:", etc.)
   - Store start/end page for each section
3. For each section:
   - Extract full text
   - Use RecursiveCharacterTextSplitter(chunk_size=512, overlap=50)
   - Mark each chunk with section metadata
4. Detect tables (look for ASCII table patterns or PDF table detection)
5. Flag table chunks with `content_type="table"` in metadata

**Key Logic:**
```
For each detected section's text:
  │
  ├─ Detect if contains table (check for | or multi-column patterns)
  │
  ├─ If table:
  │   └─ Extract as-is, mark content_type="table"
  │
  ├─ If narrative:
  │   └─ Apply RecursiveCharacterTextSplitter
  │
  └─ Add parent_id = section identifier
```

**Testing:**
- Verify 100+ chunks generated from 10-K
- Check all tables identified
- Verify no section missed
- Spot-check chunk boundaries (should end mid-sentence or at paragraph)

## Phase 2: Advanced Retrieval (1.5 hours)

### Objective
Implement hybrid retrieval (BM25 + Dense + RRF) with cross-encoder reranking for high-precision candidate selection.

### 2.1 Create BM25 Indexer

#### File: `src/retrieval/bm25_indexer.py`

**Responsibility:**
- Build inverted index for lexical search
- Enable fast keyword-based retrieval

**Input:**
- Chunk list with content
- Metadata list

**Output:**
- Pickled BM25 index saved to `indexes/bm25_index.pkl`
- Chunk ID mapping for later lookup

**Algorithm Steps:**
1. Tokenize all chunk content:
   - Lowercase all text
   - Remove punctuation (keep alphanumeric + space)
   - Split on whitespace
2. Build TF-IDF statistics
3. Store inverted index (document ID → token map)
4. Pickle entire index to disk

**Key Decisions:**
- Use `rank-bm25` library (open-source, proven)
- Min token length: 2 characters (avoid "a", "is")
- Stop words: Use English stop word list
- No stemming (preserve exact terms like "revenue" vs "earning")

**Testing:**
- Test with sample queries: "Risk Factors", "revenue", "employee"
- Verify top-5 results are relevant
- Check index size and latency (should be <5ms per query)

### 2.3 Create RRF Fusion Module

#### File: `src/retrieval/rrf_fusion.py`

**Responsibility:**
- Merge BM25 and dense search results
- Apply Reciprocal Rank Fusion scoring
- Return unified ranked list

**Input:**
- BM25 results (top-20)
- Dense results (top-20)

**Output:**
- Merged, deduped, RRF-scored top-20 results

**Algorithm (from RAG_WORKFLOW.md):**
```
For each unique document:
  If in BM25 results:
    score_bm25 = 1 / (60 + rank_bm25)
  Else:
    score_bm25 = 0

  If in dense results:
    score_dense = 1 / (60 + rank_dense)
  Else:
    score_dense = 0

  final_score = score_bm25 + score_dense

Sort by final_score descending
Return top-20
```

**Key Decisions:**
- Constant k=60 (balances both rankers equally)
- Dedup on chunk_id (exact match)
- Keep metadata from original results

**Testing:**
- Compare with BM25-only results
- Verify dense results promoted when relevant
- Check RRF prevents single ranker dominance

### 2.4 Create Cross-Encoder Reranker

#### File: `src/retrieval/cross_encoder_reranker.py`

**Responsibility:**
- Rerank top-20 candidates using semantic relevance
- Output top-3 with confidence scores

**Input:**
- Query text
- 20 candidate chunks (with metadata)

**Output:**
- Top-3 reranked results with confidence scores (0-1)

**Model Selection:**
- Model: `cross-encoder/mmarco-mMiniLMv2-L12-H384-v41` (HuggingFace)
- Size: ~70MB (download once, cache locally)
- Inference: 50ms for 20 candidates

**Algorithm Steps:**
1. Download model (first run only):
   - `from sentence_transformers import CrossEncoder`
   - `model = CrossEncoder('cross-encoder/mmarco-mMiniLMv2-L12-H384-v41')`
2. For each candidate chunk:
   - Create pair: [query, chunk_content]
3. Pass all pairs to model:
   - `scores = model.predict([pair1, pair2, ...])`
   - Returns list of scores (0-1)
4. Sort by score descending
5. Return top-3 with scores

**Testing:**
- Test query: "What are revenue figures?"
- Verify revenue table ranked first (0.95+ score)
- Verify narrative chunks with same keyword lower
- Check reranking improves precision

### 3.1 Create Query Router Module

#### File: `src/query/query_router.py`

**Responsibility:**
- Analyze user query before processing
- Classify query into routing categories to save compute
- Categories: `Simple` (direct answer/metadata), `Standard` (requires full RAG), `Multi-Hop` (complex reasoning)

**Input:**
- User query string

**Output:**
- Route classification type
- Extracted metadata filters (if any)

**Algorithm Steps:**
1. Use a fast, lightweight LLM prompt or predefined heuristic rules to classify the query.
2. Determine if the query is asking about basic document metadata -> Route `Simple`.
3. Determine if the query requires synthesizing information across multiple sections -> Route `Multi-Hop`.
4. Otherwise -> Route `Standard`.

### 3.2 Create Query Expansion Module

**Responsibility:**
- Expand user query with semantic variants
- Use synonym dictionary for financial terms

**Input:**

**Output:**

**Filtering:**

### 3.3 Create Multi-Hop Retriever

#### File: `src/query/multi_hop_retriever.py`

**Responsibility:**
- Perform initial retrieval
- Extract cross-section references
- Follow references for secondary retrieval
- Deduplicate and merge multi-hop results

**Input:**
- Query string
- Max hops (default: 2)
- Confidence threshold (default: 0.7)

**Output:**
- Top-5 results from both hops
- Tagged with hop level in metadata

**Algorithm Steps:**

**Hop 1: Initial Retrieval**
1. Run dual retrieval + RRF + reranking (from Phase 2)
2. Get top-3 results
3. Extract metadata, especially `related_sections`
4. Extract reference text from result (e.g., "See Item 8")

**Hop 2: Reference Following (if enabled)**
1. For each related_section:
   - Create new query: `original_query + " in Item X"`
   - Run dual retrieval again
   - Collect top-2 per reference (limit to 2)
2. Check confidence score > threshold before including
3. Mark each as `hop_level: 2`
4. Track `visited_sections` to prevent circularity

**Deduplication:**
- Compare chunk_id across all hops
- Keep only unique chunks (first occurrence)
- Preserve hop_level metadata

**Merging Strategy:**
1. Combine all hop results
2. Rerank using cross-encoder on full merged set
3. Return top-5 across both hops

**Testing:**
- Input: "How does geopolitical risk impact revenue?"
- Expected:
  - Hop 1: Find Risk Factors section
  - Extract: "See Item 8 for financial impact"
  - Hop 2: Retrieve Item 8 MD&A
  - Output: Both Item 1A + Item 8 in top results
  - Accuracy: 88% vs 60% (single-hop)

### 3.4 Create Corrective RAG (CRAG) Evaluator

#### File: `src/query/crag_evaluator.py`

**Responsibility:**
- Evaluate the quality and relevance of chunks retrieved before generation.
- Implement Corrective RAG (CRAG) fallback if confidence is deeply low.

**Input:**
- Original user query
- Top ranked chunks from cross-encoder

**Output:**
- Decision: `Proceed`, `Fallback`, or `Reject`
- Passed context chunks

**Algorithm Steps:**
1. Review the Top-1 cross-encoder confidence score.
2. If highest score < 0.20:
   - Identify that the system has retrieved irrelevant documents.
   - Return `Reject` ("I do not have enough context in the 10-K to answer this.") or trigger a `Fallback` (e.g., broad web search if supported).
3. If highest > 0.20:
   - Return `Proceed` with the context chunks for generation.

   - Identify citations in response

### 5.2 Create RAGAS Evaluator

#### File: `src/evaluation/ragas_evaluator.py`

**Responsibility:**
- Run system against golden dataset
- Calculate RAGAS metrics
- Generate evaluation report

**RAGAS Metrics (from RAG_WORKFLOW.md):**

1. **Faithfulness** (0-1):
   - Does answer contradict context?
   - Check: All facts in answer supported by context chunks
   - Method: LLM-based contradiction detection (simple)
   - Target: >95%

2. **Answer Relevance** (0-1):
   - Does answer address query?
   - Check: Query and answer semantic similarity
   - Method: Embedding similarity
   - Target: >90%

3. **Context Precision** (0-1):
   - Is context free from noise?
   - Check: Relevant/irrelevant chunk ratio in context
   - Method: LLM or semantic relevance check
   - Target: >85%

4. **Context Recall** (0-1):
   - Did we retrieve all needed info?
   - Check: Can we answer using retrieved context?
   - Method: Relative to expected_answer
   - Target: >90%

**RAGAS Score:**
```
RAGAS = (Faithfulness + Answer_Relevance + Context_Precision + Context_Recall) / 4
Target: 0.90+ (enterprise grade)
```

**Algorithm Steps:**
1. Load golden_questions.json
2. For each question:
   - Run `get_answer(query)`
   - Collect: answer, context_chunks, confidence
3. Calculate metrics:
   - Faithfulness: LLM check for contradictions
   - Answer Relevance: Embedding similarity
   - Context Precision: Manual or LLM review
   - Context Recall: Fuzzy match with expected_answer
4. Aggregate across all questions
5. Generate report with per-query and overall scores
